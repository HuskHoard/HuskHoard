
### Section 8: Production Deployment Guide (`production.md`)

# Production Deployment

While testing with `.img` files is useful for development, a production HuskHoard setup involves physical block devices and real-time cloud replication. This guide walk you through a professional-grade deployment.

## 1. Preparing Production Hardware

### Identifying Drives by Serial
In Linux, device paths like `/dev/sdb` can change after a reboot. For production, **always identify your drives by their hardware ID**.

Find your drive's persistent path:
```bash
ls -l /dev/disk/by-id/
```
Look for the path that includes the brand and serial number (e.g., `/dev/disk/by-id/ata-WDC_WD140EDGZ-11B1PA0_ZEK12345`).

### Initializing the Physical Drive
Unlike the test environment where we used `fallocate`, in production we format the **raw block device**. 

!!! danger "Warning: Data Loss"
    The `format` command will overwrite the beginning of the disk. Ensure you have selected the correct drive.

```bash
# Format a physical 14TB Hard Drive
sudo ./huskhoard format --tape-dev /dev/disk/by-id/ata-WDC_WD140EDGZ-11B1PA0_ZEK12345
```

## 2. Setting up the Cloud Replication Tier
HuskHoard can replicate every byte to a cloud provider in real-time.

1.  **Configure rclone:** Run `rclone config` and create a remote (e.g., named `backblaze_b2`).
2.  **Verify access:** Ensure you can see the bucket: `rclone ls backblaze_b2:my-husk-bucket`.
3.  **Add to Husk:** Use the `rclone:` prefix in your config (see step 3).

## 3. Production Configuration Example
A robust setup usually involves a "Hot" NVMe SSD, a large local "Primary" HDD, and a "Cloud" replica.

Edit `husk_config.toml`:
```toml
# --- SSD Tier ---
hot_tier = "/mnt/nvme_data/media"
db_path = "/var/lib/husk/husk_catalog.db"

# --- Hard Drive Tier (Local Archive) ---
primary_volumes = [
    "/dev/disk/by-id/ata-WDC_WD140EDGZ-SERIAL1",
    "/dev/disk/by-id/ata-WDC_WD140EDGZ-SERIAL2"
]

# --- Cloud Tier (Replication) ---
replication_volumes = ["rclone:backblaze_b2:my-husk-bucket/vault1"]
replicas = 1 # Keep 1 local copy + 1 cloud copy

# --- Production Policy ---
janitor_schedule_time = "02:00" # Run the scan every night at 2 AM
max_age_days = 60              # Keep files on SSD for 2 months
max_versions = 5               # Keep 5 historical versions for rollback
```

## 4. Running as a System Service (systemd)
In production, you want HuskHoard to start automatically when the server boots. Create a systemd unit file:

`sudo nano /etc/systemd/system/huskhoard.service`

```ini
[Unit]
Description=HuskHoard Data Tiering Daemon
After=network.target

[Service]
Type=simple
User=huskuser
Group=huskuser
WorkingDirectory=/home/huskuser/husk
ExecStart=/home/huskuser/husk/target/release/huskhoard daemon
Restart=always
RestartSec=5

# Ensure the binary has fanotify capabilities
AmbientCapabilities=CAP_SYS_ADMIN CAP_DAC_READ_SEARCH

[Install]
WantedBy=multi-user.target
```

**Enable and Start:**
```bash
sudo systemctl daemon-reload
sudo systemctl enable huskhoard
sudo systemctl start huskhoard
```

## 5. Performance Tuning for High Capacity
When dealing with multi-terabyte drives, performance tuning is key:

*   **O_DIRECT:** Production mode defaults to `O_DIRECT`. This ensures that archiving 10TB of data won't "pollute" your Linux RAM cache, leaving your system snappy for other tasks.
*   **Disk Spindown:** If you are using standard HDDs, you can set an idle spindown timer using `hdparm -S`. HuskHoard will automatically wake the drive when a "Husk" is accessed.
*   **Database Location:** Always keep your `husk_catalog.db` on your fastest SSD. This ensures that `ls` commands and file metadata lookups remain instantaneous, even if the data is on a sleeping hard drive.
