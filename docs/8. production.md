

### Section 8: Production Deployment Guide (`production.md`)

# Production Deployment

While testing with `.img` files is useful for development, a production HuskHoard setup involves physical block devices, massive multi-drive arrays, and real-time cloud replication. This guide will walk you through a professional-grade deployment.

## 1. Preparing Production Hardware

### Identifying Drives by Serial
In Linux, device paths like `/dev/sdb` can change after a reboot. For production, **always identify your drives by their hardware ID**.

Find your drive's persistent path:
```bash
ls -l /dev/disk/by-id/
```
Look for the path that includes the brand and serial number (e.g., `/dev/disk/by-id/ata-WDC_WD140EDGZ-11B1PA0_ZEK12345`).

### Initializing the Physical Drive
Unlike the test environment where we used `fallocate`, in production we format the **raw block device**. 

!!! danger "Warning: Data Loss"
    The `format` command will overwrite the beginning of the disk. Ensure you have selected the correct drive.

```bash
# Format a physical 14TB Hard Drive
sudo ./husk format --tape_dev /dev/disk/by-id/ata-WDC_WD140EDGZ-11B1PA0_ZEK12345
```

## 2. Setting up the Cloud Replication Tier
HuskHoard can replicate every byte to a cloud provider in real-time.

1.  **Configure rclone:** Run `rclone config` and create a remote (e.g., named `backblaze_b2`).
2.  **Verify access:** Ensure you can see the bucket: `rclone ls backblaze_b2:my-husk-bucket`.
3.  **Add to Husk:** Use the `rclone:` prefix in your config (see examples below).

## 3. Production Configuration Architectures
HuskHoard supports as many drives as your server can physically connect. Because of its Tiered Volume Engine, you can structure these drives for either **High Availability** or **Infinite Spillover**.

### Architecture A: High Availability Mirror (RAID 1 Equivalent)
Use this if you want absolute 1:1 local redundancy. HuskHoard multiplexes writes simultaneously.

```toml
# --- SSD Tier ---
hot_tier = "/mnt/nvme_data/media"
db_path = "/var/lib/husk/husk_catalog.db"
min_free_space_gb = 50 

# --- Tier 1: Local Primary Array ---
primary_volumes = [
    "/dev/disk/by-id/ata-WDC-SERIAL1",
    "/dev/disk/by-id/ata-WDC-SERIAL2"
]

# --- Tier 2: Real-Time Local Mirrors ---
replication_volumes = [
    "/dev/disk/by-id/ata-WDC-SERIAL3",
    "/dev/disk/by-id/ata-WDC-SERIAL4"
]
replicas = 1
```

### Architecture B: The Infinite Spillover (JBOD + Cloud)
Use this if you want maximum local capacity and use the cloud as your emergency safety net. 

```toml
hot_tier = "/mnt/nvme_data/media"
db_path = "/var/lib/husk/husk_catalog.db"
min_free_space_gb = 50 

# --- Pool all local drives for maximum capacity ---
primary_volumes = [
    "/dev/disk/by-id/ata-WDC-SERIAL1",
    "/dev/disk/by-id/ata-WDC-SERIAL2",
    "/dev/disk/by-id/ata-WDC-SERIAL3",
    "/dev/disk/by-id/ata-WDC-SERIAL4"
]

# --- Emergency Cloud Failover ---
# If all 4 local drives fill up, data seamlessly routes here
failover_volumes = ["rclone:backblaze_b2:my-husk-bucket/vault1"]

replication_volumes = []
replicas = 0
```

## 4. Understanding "Sticky Drive" Allocation
If you have 20 drives in your `primary_volumes`, HuskHoard **does not** stripe data across them like RAID 0. 

Instead, it uses a **"Sticky Drive"** algorithm. It sorts your drives by *Most Used Space First*. It will write exclusively to Drive 1 until it hits your `min_free_space_gb` safety limit. Only then will it move to Drive 2. 

**Why this matters for production:**
* **Power Savings:** Drives 2 through 20 stay spun down and sleeping in low-power mode for months.
* **Safety:** If Drive 1 dies, you only lose Drive 1. There is no catastrophic chain failure or 50-hour array rebuild required.
* **Edge-Case Safety (The Physical Fit Check):** If Drive 1 has 55GB free, and an incoming file is 100GB, HuskHoard calculates this *before* writing. It will skip Drive 1 and put the 100GB file on Drive 2, ensuring a file is never split or corrupted mid-write.

## 5. Tuning the Janitor for Batch Processing
In a test environment, it is common to use `janitor_interval_secs = 60`. In production, constant polling generates unnecessary overhead. You should transition to a nightly batch processor.

Edit `husk_config.toml`:
```toml
# Disable the testing interval
janitor_schedule_time = "02:00" # Run once a day at 2:00 AM

max_age_days = 60               # Keep files on SSD for 2 months
max_versions = 5                # Keep 5 historical versions for rollback
```

### Advanced: Tuning the Database Mirror Timeout
By default, the Archiver Worker writes a full backup of the SQLite database to the sequential archive drive whenever the queue is idle for 5 seconds. If you are doing batch processing, this creates unnecessary "wasteland" space by writing the database hundreds of times.

Before compiling for production, you can change this idle timeout to **1 Hour**:
1. Open `src/daemon.rs`.
2. Locate `match rx.recv_timeout(Duration::from_secs(5)) {`
3. Change it to `match rx.recv_timeout(Duration::from_secs(3600)) {`
4. Recompile with `cargo build --release`.

HuskHoard will now process incoming files instantly, but will wait until the system has been entirely idle for an hour before sealing the batch with a database backup.

## 6. Running as a System Service (systemd)
In production, you want HuskHoard to start automatically when the server boots. Create a systemd unit file:

`sudo nano /etc/systemd/system/huskhoard.service`

```ini
[Unit]
Description=HuskHoard Data Tiering Daemon
After=network.target

[Service]
Type=simple
User=root
WorkingDirectory=/opt/huskhoard
ExecStart=/opt/huskhoard/husk daemon
Restart=always
RestartSec=5

# Ensure the binary has fanotify capabilities
AmbientCapabilities=CAP_SYS_ADMIN CAP_DAC_READ_SEARCH

[Install]
WantedBy=multi-user.target
```

**Enable and Start:**
```bash
sudo systemctl daemon-reload
sudo systemctl enable huskhoard
sudo systemctl start huskhoard
```

## 7. LTO Tape Setup
To use a physical tape drive, ensure your user has permissions for the generic SCSI character devices.
```bash
# Add user to the tape group
sudo usermod -aG tape $USER
```
In your `husk_config.toml`, point to the **non-rewinding** device:
```toml
primary_volumes = ["/dev/nst0"] # Use 'n' prefix to prevent auto-rewind on close
```

## 8. Managing the "Offline Library"
HuskHoard excels at managing a "Shelf" of disconnected drives and tapes. 
*   **Hardware Portability:** Husk writes a UUID to LBA 0 of every volume. You can unplug a USB drive, move it to a different SATA port, and the database will automatically find it.
*   **The Recall Prompt:** If an app tries to access a file on a drive that is currently in a box on your shelf, the daemon will hold the app open and log a request via the Sidecar IPC (which you can route to Slack/Email):
    > `👉 PLEASE INSERT: Volume UUID [a1b2c3d4] (Location Hint: Box 4)`
*   **Resume:** Once the drive is plugged in, HuskHoard automatically mounts it and continues the stream.

## 9. Garbage Collection (Repacking)
Over time, as files are deleted or overwritten, old data in the archive becomes "Wasteland" (dead space). 

You can reclaim this space using the `Repack` command, which copies only the active files to a new drive:
```bash
sudo ./husk repack --source_tape /dev/sdb --dest_tape /dev/sdc
```
**How it impacts the Sticky Drive:**
If Drive 1 was full, but you Repack it so it now has 50% free space, HuskHoard **will not** immediately jump back to Drive 1. It will continue filling whatever drive is currently the *most* full (e.g., Drive 2). Drive 1 simply becomes the "backup" drive that takes over as soon as Drive 2 hits its capacity limit, preserving drive longevity.

## 10. Performance Tuning for High Capacity
When dealing with multi-terabyte drives, performance tuning is key:

*   **O_DIRECT:** Production mode defaults to `O_DIRECT`. This ensures that archiving 10TB of data won't "pollute" your Linux RAM cache, leaving your system snappy for other tasks.
*   **Disk Spindown:** If you are using standard HDDs, you can set an idle spindown timer using `hdparm -S 240 /dev/sdX` (spins down after 20 minutes of inactivity). HuskHoard will automatically wake the drive when a "Husk" is accessed.
*   **Database Location:** Always keep your `husk_catalog.db` on your fastest NVMe SSD. This ensures that `ls` commands and file metadata lookups remain instantaneous, even if the data is on a sleeping hard drive.
