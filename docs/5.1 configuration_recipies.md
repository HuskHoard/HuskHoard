

### Section 5.1 : # Configuration Recipes

HuskHoard’s policy engine is highly flexible. By tweaking your `husk_config.toml`, you can adapt the system for radically different workloads—from a simple automated tape archiver to a multi-tiered cloud media server. 

Below are several example configurations you can use as templates for your specific environment. **Note:** these snippets show only the settings that differ from the default — they aren't complete, standalone config files. Start from your auto-generated default config and merge these values in.

## 1. The Simple Tape Archiver (No Versioning, No Replication)
**Best for:** Traditional IT operators or home lab users who simply want to write files to physical LTO tape, maintain a highly-searchable SQLite catalog, and free up SSD space. No cloud, no historical versioning, and no redundant replicas.

**Key Settings:** 
* `max_versions = 1` tells the database to only keep the absolute newest version of a file and prune older records.
* `replicas = 0` entirely disables the replication logic.
* `primary_volumes` points directly to the non-rewinding tape device (`/dev/nst0`).

```toml
# --- Core Paths ---
hot_tier = "/mnt/nvme/staging_area"
db_path = "/var/lib/husk/husk_catalog.db"

# --- Hardware (Tape Only) ---
primary_volumes = ["/dev/nst0"] 
failover_volumes = []
replication_volumes = []
replicas = 0 

# --- Policy Engine ---
janitor_schedule_time = "23:00" # Run automatically once a night at 11 PM
max_age_days = 7                # Move files to tape if untouched for a week
max_versions = 1                # Disable versioning (keep only the latest version)

# Maximize tape write speed by bypassing Zstd compression on already-compressed files
no_compress_extensions = ["mp4", "mkv", "avi", "mov", "zip", "tar", "gz", "rar", "7z", "jpg", "png", "iso"]
```

## 2. Mission-Critical Backup (The "3-2-1" Strategy)
**Best for:** Enterprise data, financial records, or critical infrastructure where data loss is unacceptable. This configuration keeps local versions, creates a local failover, and pushes a replica to an offsite S3 bucket.

**Key Settings:**
* `max_versions = 10` allows you to roll back a file to 10 previous historical states.
* `replicas = 1` ensures every file written to the primary HDD is simultaneously multiplexed to the cloud.

```toml
# --- Core Paths ---
hot_tier = "/mnt/data/active_shares"
db_path = "/var/lib/husk/husk_catalog.db"

# --- Multi-Tier Volume Allocation ---
primary_volumes = ["/dev/disk/by-id/ata-WDC_WD140_Primary"]
failover_volumes = ["/dev/disk/by-id/ata-WDC_WD140_Backup"]
replication_volumes = ["rclone:aws_s3:my-company-vault/husk_backups"]
replicas = 1

# --- Policy Engine ---
janitor_schedule_time = "none"
janitor_interval_secs = 3600   # Sweep for modified files every hour
max_age_days = 1               # Archive almost immediately 
max_versions = 10              # Retain a deep historical rollback queue

# Never let the active share drive exceed 75% capacity (Emergency Spillover)
hot_tier_max_usage_percent = 75 
```

## 3. Media & Entertainment (Instant Video Offload)
**Best for:** Video editors and VFX artists who deal with massive raw footage files that rapidly consume expensive NVMe storage. 

**Key Settings:**
* `immediate_archive_dirs` and `immediate_archive_extensions` tell the Janitor to bypass the `max_age_days` rule entirely. The moment a `.braw` or `.r3d` file lands in the hot tier, it is asynchronously vaulted to cold storage and stubbed.

```toml
# --- Core Paths ---
hot_tier = "/mnt/nvme/video_projects"
db_path = "/var/lib/husk/husk_catalog.db"

# --- Volume Allocation ---
primary_volumes = ["/dev/disk/by-id/nvme-Cold_Storage_Pool"]
failover_volumes = []
replication_volumes = []
replicas = 0

# --- Policy Engine ---
janitor_schedule_time = "none"
janitor_interval_secs = 60      # Aggressive 60-second sweeps
max_age_days = 30               # Standard documents/assets stay for 30 days
max_versions = 3

# --- Aggressive Immediate Rules ---
# The moment a camera card is dumped into /Raw_Footage/, archive it instantly
immediate_archive_dirs = ["/Raw_Footage/", "/Render_Exports/"]
immediate_archive_extensions = ["braw", "r3d", "mxf", "mov", "mp4", "prores"]
no_compress_extensions = ["braw", "r3d", "mxf", "mov", "mp4"]
```

# Maintenance & Disaster Recovery

HuskHoard is built with a "Data-First" mentality. We assume that SSDs will fail, databases will get corrupted, and bit-rot is inevitable. These tools are designed to protect you from those realities.

## 1. Bit-Rot Protection (The Scrubber)
Over years of storage, magnetic bits on a hard drive can "flip," causing silent data corruption. HuskHoard stops this by cryptographically signing every object with a **BLAKE3 hash**.

Run the scrubber periodically (e.g., once a month) to verify your archive:
```bash
./huskhoard scrub --tape-dev /dev/sdb
```
The Scrubber will:
1. Read every header and verify the CRC32 checksum.
2. Decompress the data stream in memory.
3. Re-calculate the BLAKE3 hash and compare it to the original.
4. Report exactly which files (if any) have been corrupted.

## 2. Reclaiming Space (The Repacker)
Because HuskHoard uses a **Log-Structured** format, it never deletes data from the middle of a volume (which would be slow and wear out the drive). Instead, it just marks old versions as "Wasteland."

When your `info` command shows a high percentage of **Reclaimable Space**, run the Repacker:
```bash
./huskhoard repack --source-tape /old_drive.img --dest-tape /new_drive.img
```
The Repacker performs "Garbage Collection":
* It identifies only the **latest active version** of every file on the source.
* It copies those files to the destination.
* It updates the Catalog to point to the new offsets.
* The old drive can then be wiped and reused.

## 2.5 Catalog Reconciliation (The Pruner)
If your users frequently delete files via Windows SMB/standard `rm` and expect them to be permanently destroyed (rather than "Deep Archived"), your tape library will eventually fill up with "orphaned" files. 

You can force the Catalog to sync perfectly with your SSD by running the Pruner:
```bash
./huskhoard prune
```
The Pruner scans the entire database. If a file exists in the database but is completely missing from the SSD (no stub, no file), it permanently deletes the record from the Catalog. 
*Note: Run a `repack` after a massive `prune` to physically free the space on your tapes.*

## 3. Disaster Recovery (The Rebuild)
If your SSD dies and you lose your `husk_catalog.db`, do not panic. Your data is not lost.
*   **For Disks:** HuskHoard scans for "USTD" magic bytes across the LBA range.
*   **For Tapes:** HuskHoard reads every hardware Filemark, extracts the 4KB header at the start of each file, and re-indexes the tape's contents into a new SQLite database.

Because every 4KB header on the archive volume contains the filename and metadata, HuskHoard can perform a **Deep Scan** to rebuild your database from scratch:
```bash
./huskhoard rebuild --tape-dev /dev/sdb --output-db husk_recovered.db
```
HuskHoard will scan the entire physical surface of the drive, identify "USTD" magic bytes, and re-index every file it finds.

## 4. The "Easy Exit" (Manual Extraction)
We believe you should never be locked into a storage tool. If you need to get your data out and don't want to use HuskHoard, you can use standard Linux tools:
1. Use `dd` or a hex editor to find the offset of a file (skipping the 4096-byte header).
2. Pipe the raw data into the standard `zstd` utility:
   `dd if=/dev/sdb bs=4096 skip=[offset+1] | zstd -d > recovered_file`
