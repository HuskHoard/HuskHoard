# Architecture

HuskHoard is built as a multi-threaded daemon that sits between the Linux Kernel and your storage hardware.

## The Lifecycle of a File
A file managed by HuskHoard moves through four distinct states:

1.  **HOT:** The file lives entirely on your SSD.
2.  **COOLING:** The **Janitor** monitors the file. If it isn't touched for `max_age_days`, it is queued for archiving.
3.  **ARCHIVING:** The **Worker** streams the file, compresses it, and replicates it across your storage pool.
4.  **STUBBED (The Husk):** The file's blocks are freed on the SSD. It occupies 0 bytes of physical space but remains 100% visible to the OS.

## Core Components

### 0. The HTTP Gateway (VLC / Plex Bridge)
The Gateway is a lightweight, zero-copy HTTP/1.1 server that listens on `127.0.0.1`. It translates standard HTTP Range requests from media players into StreamGate seek commands. This allows HuskHoard to act as a local "Cloud Provider" for media servers without using a FUSE mount.

### 1. The Interceptor (fanotify)
Uses the Linux `fanotify` kernel API to block processes attempting to read "Stubbed" files. It triggers the **Rehydration** engine and releases the process once the data is available.

#### 2. StreamGate: The Indexed Direct Pipe
Unlike standard archivers that require you to decompress a file from the beginning, HuskHoard implements **StreamGate**. 
*   **Frame-Based Compression:** Files are compressed into independent 16MB Zstd frames.
*   **The Jump Table:** During the archival process, HuskHoard records the exact byte-offset of every 16MB frame in the SQLite catalog.
*   **O(1) Seeking:** When you request a partial read (e.g., via `cat --offset`), HuskHoard calculates which frame contains that byte, seeks the hardware (Tape/Disk) or Cloud stream directly to that frame, and begins decoding. This enables instant seeking in massive media files.

### 3. The Archive Worker (SCSI & Cloud Driver)
The Worker manages the hardware interface:
*   **LTO Tape Driver:** Sends SCSI commands (`Rewind`, `Space to End`, `Filemark`) to physical tape drives.
*   **SCSI Block Driver:** Optimized 256KB-aligned writes for hardware compatibility.
*   **Cloud Driver:** Multiplexed streaming to `rclone` remotes.

### 4. The Catalog (SQLite)
The Catalog is the "Brain." It tracks every version of every file and its exact byte-offset on your "tapes." 
> **Resilience Note:** The Catalog is periodically backed up to the archive volumes themselves. This creates a "Database Anchor," allowing for total disaster recovery even if the host SSD is wiped.

#### 5. Disaster Recovery: The "Database Anchor"
HuskHoard follows the **Easy Exit** philosophy. You are never locked into the database.
*   **Catalog Mirroring:** After every archive session, the SQLite database is vacuumed and mirrored onto the primary archive volumes themselves. 
*   **Deep Scan Rebuild:** In the event of a total host SSD failure, HuskHoard can perform a "Deep Scan" of any tape or archive disk. It reads the 4KB Object Headers directly from the media to reconstruct a perfect, version-accurate SQLite catalog from scratch.
