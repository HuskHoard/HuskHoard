# Maintenance & Disaster Recovery

HuskHoard is built with a "Data-First" mentality. We assume that SSDs will fail, databases will get corrupted, and bit-rot is inevitable. These tools are designed to protect you from those realities.

## 1. Bit-Rot Protection (The Scrubber)
Over years of storage, magnetic bits on a hard drive can "flip," causing silent data corruption. HuskHoard stops this by cryptographically signing every object with a **BLAKE3 hash**.

Run the scrubber periodically (e.g., once a month) to verify your archive:
```bash
./huskhoard scrub --tape-dev /dev/sdb
```
The Scrubber will:
1. Read every header and verify the CRC32 checksum.
2. Decompress the data stream in memory.
3. Re-calculate the BLAKE3 hash and compare it to the original.
4. Report exactly which files (if any) have been corrupted.

## 2. Reclaiming Space (The Repacker)
Because HuskHoard uses a **Log-Structured** format, it never deletes data from the middle of a volume (which would be slow and wear out the drive). Instead, it just marks old versions as "Wasteland."

When your `info` command shows a high percentage of **Reclaimable Space**, run the Repacker:
```bash
./huskhoard repack --source-tape /old_drive.img --dest-tape /new_drive.img
```
The Repacker performs "Garbage Collection":
* It identifies only the **latest active version** of every file on the source.
* It copies those files to the destination.
* It updates the Catalog to point to the new offsets.
* The old drive can then be wiped and reused.

## 3. Disaster Recovery (The Rebuild)
If your SSD dies and you lose your `husk_catalog.db`, do not panic. Your data is not lost.
*   **For Disks:** HuskHoard scans for "USTD" magic bytes across the LBA range.
*   **For Tapes:** HuskHoard reads every hardware Filemark, extracts the 4KB header at the start of each file, and re-indexes the tape's contents into a new SQLite database.

Because every 4KB header on the archive volume contains the filename and metadata, HuskHoard can perform a **Deep Scan** to rebuild your database from scratch:
```bash
./huskhoard rebuild --tape-dev /dev/sdb --output-db husk_recovered.db
```
HuskHoard will scan the entire physical surface of the drive, identify "USTD" magic bytes, and re-index every file it finds.

## 4. The "Easy Exit" (Manual Extraction)
We believe you should never be locked into a storage tool. If you need to get your data out and don't want to use HuskHoard, you can use standard Linux tools:
1. Use `dd` or a hex editor to find the offset of a file (skipping the 4096-byte header).
2. Pipe the raw data into the standard `zstd` utility:
   `dd if=/dev/sdb bs=4096 skip=[offset+1] | zstd -d > recovered_file`

## 1. Physical Tape Maintenance
When using physical LTO tapes, the **Scrubber** and **Repacker** currently require the volume to be on a standard Disk/Image. Support for native sequential scrubbing of LTO hardware is currently in the roadmap.
