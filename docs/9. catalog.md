
### Section 9: The Catalog (SQLite Deep Dive) (`catalog.md`)

# The Catalog Database

HuskHoard follows the **"Open Data"** principle. All file metadata, version history, and hardware tracking are stored in a standard **SQLite 3** database. We don't hide your information in proprietary binary blobs.

## 1. Locating the Database
The path to your catalog is defined in your `husk_config.toml`:
```toml
db_path = "husk_catalog.db"
```
HuskHoard uses **WAL (Write-Ahead Logging)** mode for performance. You may see two extra files (`-shm` and `-wal`) alongside your database while the daemon is running. Do not delete these!

## 2. Viewing the Database
Since it is a standard SQLite file, you can open it with any standard tool:
*   **CLI:** `sqlite3 husk_catalog.db`
*   **GUI (Recommended):** [DB Browser for SQLite](https://sqlitebrowser.org/) (Available on Linux, Mac, and Windows).

## 3. The Schema (Where the Data Lives)
The database consists of three primary tables:

### `catalog`
This is the master index of every file version.
*   `file_path`: The original path on your SSD.
*   `version`: Incremental version number (for Point-in-Time Recovery).
*   `tape_uuid`: Which volume contains this file.
*   `tape_offset`: The exact byte-offset on the volume where the file starts.
*   `blake3_hash`: The cryptographic fingerprint of the file.

### `tapes`
The registry of all physical and virtual archive volumes.
*   `tape_uuid`: Unique identifier.
*   `device_path`: Last known Linux path (e.g., `/dev/sdb`).
*   `drive_serial`: The hardware serial number used for the **Offline Recall** prompts.

### `active_tracking`
The "Cooling Room." This tracks files on your SSD that the daemon is currently monitoring for age/access.

## 4. Useful SQL Queries
You can run these commands via `sqlite3` to get deep insights into your Hoard:

**Find the 10 Largest Files in the Archive:**
```sql
SELECT file_path, payload_size / 1024 / 1024 AS MB 
FROM catalog 
ORDER BY payload_size DESC LIMIT 10;
```

**Check which Physical Drive a file is on:**
```sql
SELECT c.file_path, t.device_path, t.drive_serial 
FROM catalog c 
JOIN tapes t ON c.tape_uuid = t.tape_uuid 
WHERE c.file_path LIKE '%my_video.mp4%';
```

**Calculate Total Space Saved on SSD:**
```sql
SELECT SUM(payload_size) / 1024 / 1024 / 1024 AS GB_SAVED FROM catalog;
```

## 5. The "Easy Exit" Recovery (Manual)
If you ever want to leave HuskHoard or recover a file manually, the database gives you the "treasure map."

1.  **Get the Map:** Run `SELECT tape_offset, compressed_size FROM catalog WHERE file_path = 'your_file.txt'`.
2.  **Locate the Tape:** Find the physical drive matching the `tape_uuid`.
3.  **Extract with Standard Tools:**
    Since HuskHoard uses 4096-byte headers and standard Zstd streams, you can extract your data using only the `dd` and `zstd` commands:
    ```bash
    # Skip the 4KB header and extract the payload
    dd if=/dev/sdb bs=4096 skip=([tape_offset / 4096] + 1) count=[compressed_size / 4096 + 1] | zstd -d > recovered_file.txt
    ```

## 6. Database Backups
Because the database is the "Brain" of the system, HuskHoard performs **Database Anchoring**:
*   Every time the Archive Worker finishes a batch, it performs a `VACUUM INTO` to create a clean backup.
*   That backup is then **archived onto the tape itself**.
*   This ensures that even if your computer and its SSD are destroyed in a fire, as long as you have your archive drives, you can recover the Catalog and every file within it.
