
### Section 10: High-Performance Search (`search.md`)

# High-Performance Search

As your "Hoard" grows to millions of files and multiple petabytes, traditional Linux search tools like `find` or `locate` become painfully slow. HuskHoard solves this by providing **Instant Metadata Search** via the Catalog.

## 1. Indexed vs. Recursive Search
*   **The Old Way (`find`):** To find a file, the OS must recursively traverse every directory. This involves thousands of random I/O operations. On a cold hard drive, this can take minutes.
*   **The Husk Way (SQL):** HuskHoard maintains a **B-Tree Index** on the `file_path` and `blake3_hash` columns. Search results are returned in **milliseconds**, regardless of whether the archive drives are even plugged in.

## 2. Searching the Hoard
You can perform complex, high-speed queries that are impossible with standard filesystem tools.

### Instant Path Search
Find every file in your archive with "2015" in the name, across all volumes:
```sql
-- Results in < 0.01 seconds
SELECT file_path, version FROM catalog WHERE file_path LIKE '%2015%';
```

### Searching by Bit-Rot Fingerprint (BLAKE3)
HuskHoard allows you to search by the **content**, not just the name. If you have a file and want to know if it already exists in your archive (even if renamed), you can search by its BLAKE3 hash:
```sql
-- Find duplicates or renamed versions instantly
SELECT file_path, tape_uuid FROM catalog WHERE blake3_hash = 'd37a...8b2f';
```

### Metadata & Ownership Search
Find all files belonging to a specific user (UID 1000) that are larger than 1GB:
```sql
SELECT file_path, payload_size / 1024 / 1024 / 1024 AS GB 
FROM catalog 
WHERE uid = 1000 AND payload_size > 1073741824;
```

## 3. "Ghost" Search (Searching Offline Volumes)
This is the most powerful aspect of the HuskHoard Catalog. 

Standard Linux search tools can only see files that are currently "online." If you unplug a USB drive, `find` can no longer see its contents. 
**HuskHoard remembers everything.** 

Because the Catalog is stored on your SSD, you can search your entire library—including drives sitting on your shelf—without plugging them in. Once you find the file you need, HuskHoard will tell you exactly which drive serial number to go grab.

## 4. Future Tooling: `husk search`
While you can currently search using the `sqlite3` CLI, the HuskHoard roadmap includes a dedicated `husk search` command to make this even easier:

```bash
# [Planned Feature]
./huskhoard search "*.mkv" --min-size 10G --offline
```

## 5. Integration with External Tools
Because the search is so fast and the database is standard, you can point other tools at the HuskHoard Catalog:
*   **Python:** Build a web-based "File Gallery" by querying the SQLite DB.
*   **Reporting:** Generate a weekly "Storage Report" showing which users are consuming the most archive space.
*   **Security:** Scan the `blake3_hash` list against known malware databases without ever having to "read" the files from the slow hard drives.
