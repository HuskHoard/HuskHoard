## Section 5: User Guides (`usage.md`)

# User Guides

Once the HuskHoard daemon is running, it operates largely in the background. However, you have full control over the archival policies and recovery process.

## 1. Controlling Archival Policies
Archive behavior is defined in `husk_config.toml`.

### Scheduling the Janitor
You can control exactly when HuskHoard consumes system resources to archive data. For production, schedule it to run at night. For testing, use an interval.
```toml
# Run once a day at 2:00 AM
janitor_schedule_time = "02:00" 
```

Or set to "none" to run continuously based on an interval
```toml
janitor_schedule_time = "none"
janitor_interval_secs = 60
```
### Age-Based Archiving
The most common use case is archiving files once they haven't been touched for a certain period.

```Toml
max_age_days = 30 # Files not accessed in 30 days become Husks
```
### Exclusions & Temporary Files
You can prevent certain directories or temporary files from ever being archived or tracked:

```Toml
exclude_dirs = ["/.git/", "/node_modules/", "/__pycache__/"]
temp_extensions = [".swp", ".tmp", ".crdownload", "~", ".part"]
```
### High-Performance Raw Copy (Compression Bypass)
Many media formats (MP4, MKV) and archives (ZIP) are already highly compressed. Running Zstd on them wastes CPU cycles and slows down archiving. HuskHoard allows you to bypass compression for these extensions, resulting in a direct, high-speed raw copy to the archive:

```Toml
no_compress_extensions = ["mp4", "mkv", "avi", "mov", "zip", "tar", "gz", "rar", "7z", "jpg", "png", "iso"]
```

### Immediate Archiving
Some files (like raw video footage or ISOs) should never take up SSD space. You can force these to archive as soon as the Janitor sees them:
```toml
immediate_archive_extensions = ["mp4", "mkv", "iso", "zip"]
immediate_archive_dirs = ["/ArchiveDrop/"]
```
### Emergency Spillover & Capacity Management
The High-Water Mark Spillover feature serves as a critical safety valve for the Hot Tier (SSD). While the standard policy engine archives files based on age (e.g., max_age_days), the Spillover logic monitors the physical reality of your disk space. During every Janitor scan, HuskHoard calculates the current disk usage of the Hot Tier. If the usage percentage exceeds the hot_tier_max_usage_percent defined in your configuration, the system enters "Emergency Spillover" mode.
In this state, HuskHoard calculates exactly how many bytes must be freed to return to the target threshold. It then queries the catalog for un-stubbed files, sorted by the oldest access time (LRU). These files are moved to the storage pool and stubbed immediately, overriding any minimum age requirements. This ensures that your high-speed storage remains available for new ingest and system operations, even during unexpected bursts of data growth.
```toml
# Trigger emergency archiving if the Hot Tier exceeds 80% capacity
hot_tier_max_usage_percent = 80 

# The Janitor will try to keep at least this much space (in GB) strictly free
min_free_space_gb = 5
```
## 2. Restoring Data
HuskHoard provides two ways to get your data back.

### A. Transparent Rehydration (The "Just Open It" Method)
Simply open the file in any application (VLC, Image Viewer, Text Editor). HuskHoard will intercept the read, fetch the data from the Hoard, and fill the "Hole" on your SSD. The application will continue as if the file was always there.

> **Fast-Path Bypass (Zero-Restore Overwrites):** If an application opens a stubbed file strictly to truncate and overwrite it (e.g., rendering a new video export over the old one), HuskHoard detects the `O_TRUNC` flag. It drops the stub status and allows the overwrite instantly **without** pulling the old data from tape, saving massive amounts of time and I/O.

### B. Manual CLI Restore (PITR)
HuskHoard maintains a history of your files. If you accidentally overwrite a file and want the version from three days ago, use the `restore` command:
```bash
./huskhoard restore --file-path /home/jo/report.pdf --dest-path ./recovered.pdf --version 1
You can also specify exactly which replica volume to restore from using the --source parameter (useful if your primary volume is temporarily offline but a cloud or secondary replica is available):
```bash
./huskhoard restore --file-path /home/jo/report.pdf --dest-path ./recovered.pdf --source "rclone:backblaze_b2"
```

## 3. Monitoring Volume Health
Use the `info` command to see the "Tank Gauge" of your storage volumes.
```bash
./huskhoard info --tape-dev /dev/sdb
```
This will display:
*   **Physical Usage:** How much of the drive is physically written.
*   **Active Data:** How much of that data represents current files.
*   **Reclaimable Space:** Space occupied by old versions or deleted files that can be recovered by running a **Repack**.

## 4. Understanding Log Levels
If you are troubleshooting, change the `log_level` in your config:
*   `error`: Only show critical failures.
*   `info`: Default. Shows archival and restore activity.
*   `debug`: Shows detailed fanotify events and database queries.

#### 5. StreamGate: Direct Access vs. Rehydration
HuskHoard offers two distinct ways to access your archived data:

| Feature | **Transparent Rehydration** | **StreamGate (CLI)** |
|:---|:---|:---|
| **Trigger** | Double-click or `cp` the file | `huskhoard cat` |
| **SSD Space** | File is written back to SSD | **Zero SSD space used** |
| **Best For** | Editing files, small documents | 4K Video, Log analysis, ISOs |

#### 6. Partial Reads & Seeking (StreamGate)
Because HuskHoard uses indexed 16MB Zstd frames, you can jump to specific parts of massive files instantly.

**Extract the last 500MB of a 50GB Archive:**
```bash
# Calculate your offset and pipe to a local file
./huskhoard cat --file-path big_backup.tar --offset 49500000000 > tail_fragment.tar
```

**Watch a movie starting at the 1-hour mark:**
```bash
# Assuming 1GB roughly equals 30 mins for high-bitrate video
./huskhoard cat --file-path movie.mkv --offset 2000000000 | mpv -
```

## 7. Deleting Data (Deep Archive vs. True Delete)
HuskHoard offers two completely different workflows for deleting files depending on your retention needs:

### A. Deep Archiving (Standard OS Delete)
If you delete a file using standard OS tools (e.g., `rm`, deleting via a Windows SMB share, or moving to the Trash), the file is removed from your SSD. **However, it remains safely in the HuskHoard Catalog and on your physical tapes.**
* **Use Case:** You are done working on a project and want it off your active NVMe drive, but you still want it archived forever on tape.

### B. True Delete (Permanent Destruction)
If you have a file that is garbage (e.g., a bad video take, corrupted render) and you want it completely purged from both the SSD and the underlying Tape database, you must use the Husk CLI:
```bash
./huskhoard rm --file-path /home/jm/bad_render.mp4
```
* **Use Case:** Reclaiming tape space and permanently destroying sensitive or useless files. *Note: The physical space on the tape is fully reclaimed the next time you run a `repack`.*
